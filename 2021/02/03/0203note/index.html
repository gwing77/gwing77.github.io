分布式存储

读写分离再业务中的落地

互联网行业大部分业务都是读多写少的，为了不让数据库的读取工作成为业务瓶颈，保证写库的成功率，一般会采取读写分离的技术。

  • 主数据库增删改
  • 从数据库处理查询语句

把访问压力转移到从库，与此同时，对数据库也提出了一些要求，数据一致性，确保不同的操作指向不同的库。

方案1：应用层解决

优点：数据源切换方便，可以由程序自动完成；不需要引入中间件；理论上可以支持任何数据库
缺点：由开发人员完成，运维人员参与不到

方案2：中间件解决

优点：源程序不需要做任何改动就可以实现读写分离；动态添加数据源不需要重启程序
缺点：程序依赖中间件，会导致切换数据库变得困难；由中间件做中转代理，性能会有所下降

MYSQL的主从复制

读写分离是基于主从复制架构实现的，主从复制是通过二进制日志binlog实现的
主从复制下的延时问题

  • 敏感业务强制读取主库
  • 关键业务不进行读写分离

如何避免丢数据

  • 异步复制（默认，性能最好，但是容易不一致）
  • 半同步复制
  • 全同步复制（所有从库同步完了才给用户返回，牺牲性能，保证一致性）

为什么要分库分表

why？

  • 读写的数据量限制单表行数超过500万或者单表通量超过2GB时，推荐进行分库分表
  • 数据库连接限制，数据库的连接时有限制的
  • 避免热点数据的访问都集中在单点机器上

分库分表的原理

垂直切分（按照字段切分）和水平切分（按照记录切分）

分库分表要注意的问题

分布式事务的问题：可以使用分布式事务中间件，实现TCC等事务模型；也可使用基于本地消息表的分布式事务实现
跨库关联查询问题：跨库跨表的查询操作实现起来会比较复杂，性能也无法保证
跨库跨表的合并和排序问题：需要依赖开源的分库分表中间件处理

中间件实现

业务实现分库分表的话，需要自己实现路由规则，实现跨库合并排序等操作，具有一定的开发成本，可以考虑使用开源的分库分表中间件。淘宝TDDL和Apache shardingSphere

分库分表后，如何解决主键唯一的问题

  • 使用单点数据表:每次要生成唯一ID的时候，就去这张数据表里面新增一条记录
  • 使用UUID，java中内置了UUID的工具类实现,UUID由MAC地址，时间序列，当前日期时间相关
    1. 1~8位采用系统时间，在系统时间上精确到毫秒级保证时间上的惟一性；
    2. 9~16位采用底层的IP地址，在服务器集群中的惟一性；
    3. 17~24位采用当前对象的HashCode值，在一个内部对象上的惟一性；
    4. 25~32位采用调用方法的一个随机数，在一个对象内的毫秒级的惟一性。
  • 基于snowflake算法：Twitter的开源分布式ID生成算法，由64为的二进制数字组成，一共为四部分

    第一位默认就试0，保证数值是正整数
    41位毫秒时间戳，能表示69年多一点，足够业务中使用
    10位ID是1024个节点
    12位序列号作为当前时间戳和机器下的流水号，每毫秒可以有4096个ID，能承受409万的QPS

snowflake算法可以作为一个单独的服务，部署在多台机器上，产生的ID是趋势递增的
snowflake需要注意的是需要通过NTP网络时间协议，同步各个计算机的时间，否则可能会出现重复ID

  • 基于数据库维护区间分配，创建sequence表，记录每个业务主键当前已被占用的ID区间的最大值，需要插入记录的时候，取对应的ID区间缓存在本地，同时更新sequence表
    在本地可以用原子类进行ID分配
    sequence表所在的数据库通常会配置多个从库，实现高可用
  • 几种主键方案的总结
    1. 主键必须唯一
    2. 需要满足有序性，逐渐单调递增或满足时间段内的递增
    3. 要能尽快生成主键，同时满足高可用

分库分表后，如何进行扩容

  1. 存储实现：避免数据丢失，要求数据一致性，选择支持事务的关系型数据库，如mysql和innoDB存储引擎
  2. 数据库高可用，配置主从复制读写分离，设置多个从库，提高数据安全
  3. 业务量足够大需要分库分表
  4. 考虑路由规则与扩容方案
    • 对主键进行哈希取模：根据不同的业务主键输入，对数据库取模，得到插入数据的位置 ，数据拆分均匀，但是不利于后面的扩容
    • 基于数据范围进行路由，根据特定的字段进行划分不同的区间，例如订单表可以按照ID的大小分不同的表，扩容时可以直接增加新的存储，缺点是数据访问不均匀，丧失了分库分表的优势
    • 结合哈希和数据范围的分库分表规则，比较好的平均，既能扩容，数据又平均分散
    • 预定义算法规则，事先规划好具体的库和表的数量，然后直接指定库或表

nosql数据库的应用

SQL语言的全称是structured query language，结构化查询语言，对应的存储实现是关系型数据库
nosql数据库可以理解为not only SQL,也可以认为是non-relational SQL

nosql数据库的特点

  • 良好的扩展性，容易通过集群部署：与sql不同nosql数据库去掉了关系型数据库的关系特性，天生集群友好
  • 读写性能高，支持大数据量：nosql采用全内存实现，更适合高并发的访问场景
  • 不限制表结构，灵活的数据模型：nosql不需要实现为数据建立存储结构和字段，可以存储各种类型的自定义数据

数据库特性

实际工作场景中，两种数据库之间是相辅相成的

  • nosql数据库关注的是AP模型，数据高性能
  • sql数据库强调的是CP模型，数据可靠性

nosql数据库应用

  • key-value数据库：实现缓存等应用，redis&memcached
  • 文档型数据库：存储结构化的文档，mongoDB
  • 列存储数据库：hbase，存储海量的数据

shardingSphere

内容

shardingSphere是一种关系型数据库中间件

  • sharding-JDBC，是shardingSphere的前身，轻量级java框架，再JDBC层提供扩展性服务，对外暴露的接口和JDBC规范中提供的接口完全一致,jar包形式提供服务，直接连接数据库，无需额外的部署和依赖管理
  • sharding-proxy，透明化的数据库代理端，提供封装了二进制协议的服务端版本，支持异构语言，可以像使用一个简单的数据库一样来操作分库分表的数据

对比前两者的不同：可以被混在一起使用，sharding-jdbc实在应用程序中直接嵌入jar包，适用于业务开发人员；sharding-proxy提供静态入口和异构语言支持，适合对分片数据库管理的中间件开发和运维人员

  • sharding-sideCar，目标是把系统中各种异构的服务组件串联起来，并进行高效的服务治理

四大核心功能

  1. 基础设施
    • 微内核架构：设计上采用微内核架构模式，内核系统+插件，高度可扩展
    • 分布式主键：默认采用雪花算法
  2. 分片引擎
    • 数据分片：核心功能，支持垂直拆分和水平拆分
    • 读写分离：基于主从数据库架构的读写分离机制他，可以和数据分片完美进行整合
  3. 分布式事务
    • 标准化事务处理接口：支持本地事务/基于XA两端提交的强一致性事务/基于BASE的柔性最终一致性事务，同时抽象了一组标准化事务处理接口
    • 强一致性事务与柔性事务：atomikos/narayana/bitronix实现XA事务管理器，整合seata提供柔性事务功能
  4. 治理和集成
    • 数据脱敏：保证数据访问安全，shardingSphere的强大之处，将SQL改写加密内嵌到SQL执行过程中，业务人员不需要关注具体的加解密关键实现，而只用通过简单的配置就能实现数据的自动脱敏
    • 配置中心：支持动态管理配置信息，数据源/表与分片及读写分离策略的动态切换
    • 注册中心：提供了基于nacos和zk的两种方式
    • 链路跟踪
    • 系统集成：与spring框架的集成

JDBC规范与shardingSphere之间的关系

基于适配器模式的JDBC重写实现方案
shardingSphere是一种典型的客户端分片解决方案，而客户端分片的实现方式之一就是重写JDBC规范

shardingSphere支持行表达式

行表达式是用于实现简化和统一配置信息的一种工具，在日常开发过程中应用得非常广泛
