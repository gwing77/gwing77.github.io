分布式技术：计算引擎

一、大数据导读

了解Doug Cutting，在家里设计了第一个提供全文文本搜索的开源数据库Lucene，目的是为了搜搜菜谱，后来演变成Hadoop。
从hadoop成功学到经验

  1. 代码开源：社群的力量让它成为最好的搜索技术
  2. “数字转型”需要不同的运算和储存架构，数字转型指ETL数据提取转换和加载
  3. hadoop与机器学习、AI有高度关联性，训练，测试，评价人工智能都需要数据，可以用这个平台搜集各种巨量数据

了解数据从日志开始

  • 日志是什么？
    1. 数据库中的日志是立即持久化的，发生崩溃时用于恢复其它持久化结构
    2. 分布式系统中的日志解决了两个问题：更改动作的排序和数据的并发
    3. 变更日志101：表和事件的二象性，表支持了静态数据，日志记录了变更
    • 数据集成，让组织中所有存储和处理系统可以容易地访问组织所有的数据
      1. 事件数据的爆发：增长事件的数据，往往比传统数据库应用大好几个数量及
      2. 数据系统的爆发：专门用于OLAP（联机分析处理）、搜索、简单在线存储、批处理、图像分析等的数据系统开始流行，并可以免费获得
    • 流处理，持续数据流处理的基础设施
      1. 处理模型的真正驱动力是数据收集方法，当数据是持续收集的，自然也应该持续不断的处理。
    • 系统构建：数据基础结构不是打包套餐而是分解单品成一组服务及面向应用的API
    • 日志能做什么？
      • 通过对节点并发更新的排序处理，处理数据的一致性
      • 提供节点之间的数据复制
      • 为写入者提供commit语义，确保写入不会丢失时才会收到完成确认
      • 为系统提供外部的数据订阅
      • 对于丢失数据的失败了的副本，提供恢复或是重新启动一个新的副本的能力
      • 调整节点之间的数据平衡

二、大数据生态

Hadoop

Hadoop主要由三部分组成，集中管理，分布存储与计算

  • 分布式文件系统 HDFS：分布式文件存储是分布式计算的基础，也可见分布式文件存储的重要性
    • 文件数据以数据块的方式进行拆分
    • MapReduce对输入数据进行分片读取，可以启用多个进程对多个数据块进行并发访问
    • DataNode存储的数据块会进行复制，每隔数据块存在多个备份，保证可靠性
  • 分布式计算框架 MapReduce：开发人员必须基于MapReduce编程模型进行编程开发，然后将计算框架分发到Hadoop集群中运行
    • 需要做的事情是编写一个map函数和一个reduce函数，函数是如何分布启动到集群上，也不用关心数据块是如何分配给计算任务的
    • mapreduce只能处理可分割的任务，大数据场景几乎都是可以分片计算的
  • 分布式集群资源调度框架 Yarn：
    • Hadoop早期服务器集群调度和mapreduce执行过程耦合在一起，后来出现了很多大数据框架，yarn被分离出来
    • 大数据框架都要遵守yarn规范的applicationMaster，这样一个集群中就可以同时并发执行不同的大数据计算框架

Hive

在数据仓库中，sql是最常用的分析工具，sql可以通过MapReduce实现，Hive能把SQL转化成MapReduce能执行的代码

Spark

Spark比MapReduce的计算速度快100多倍。
MapReduce是面向过程的大数据计算，spark是面向对象的大数据计算。spark是针对数据进行编程，将大数据几何抽象成一个RDD对象，然后在RDD上进行计算处理

Hbase

Hbase是GoogleBigTable的开源实现，是完整继承了BigTable的优良设计的nosql系统

  • 架构上通过数据分片的设计配合HDFS，实现了数据的分布式海量存储
  • 数据结构上通过列族设计，实现了数据表结构可以在运行期定义
  • 存储上使用LSM树（Log-Structured Merge-Tree），使数据可以通过连续写磁盘的方式保存数据，提高数据写入性能
    Hbase的海量数据处理能力，根本在于和传统关系型数据库设计的不同思路，nosql数据库认为，业务逻辑就应该由程序去处理，简单暴力也是一种美

Storm、Spark Streaming、Flink

前面介绍的大数据技术主要是处理、计算介质上的大规模数据，这类计算也叫作大数据批处理计算，这类计算一般使用mapreduce或spark这样的计算框架，一般完成一次计算要几分钟到几小时
还有一种大数据处理技术是针对实时产生的大规模数据进行即时计算处理，数据不是存储在HDFS上，而是实时传输过来的，成为大数据流计算系统

Flink

大数据引擎发展的几个阶段

  • 海粮油节数据 Apache Hadoop
  • 海量无界数据 Apache Storm
    前两个时期一般用户会维护两套系统，批处理要求低的有限数据集，流系统处理延迟要求较高的实时数据
  • 一套系统处理有节无界数据：Apache Spark，批处理系统，micro-batch方式处理无界数据
  • Flink的出现：spark在许多低延迟的场景下还是无法胜任，flink是一个以流为核心的批流一体计算引擎，高吞吐，低延迟

大数据技术分类

  • 存储：文件存储HDFS、列存储Hbase（Hbase底层存储也用到了HDFS）
  • 计算：批计算MapReduce、Spark；流计算Spark Streaming、Storm、Flink；数据仓库Hive、Spark SQL、Flink SQL
  • 资源管理：Yarn
    MapReduce、Spark、Hive、Spark SQL主要用来解决离线大数据的计算，批处理计算
    Storm、Flink、Spark Streaming针对实时数据进行计算，流处理大数据计算

数据仓库

数据仓库是一个面向主题的、集成的、非易失的且随时间变化的数据集合，用来支持管理人员的决策。

  • 离线数仓：存放数据的仓库，偏分析(Hive)，结果不及时(t+N)
  • 实时数仓：存放数据的仓库，偏分析（Flink），结果不及时（T+0）

三、Flink基础

Flink架构模型

  • Flink分层：SQL—Table Api（前面这两个是更高级的API操作，用于降低用户使用实时计算的门槛）—DataStream、DataSet Api（开发者主要针对core Api编程）—Stateful Stream Processing
  • Flink的数据流模型（Streams与Transformations）
    1. 每一个数据流起始于一个或多个Source
    2. 经过Transformation转换部分Map、Keyby、TimeWindow等算子进行逻辑运算，经常会包含多个转换算子
    3. 最后，经过计算的数据会被写入到我们执行的文件中，这便是我们的Sink部分
  • Flink核心概念窗口和时间
    1. 对数据流上的聚合需要由窗口来划定范围
    2. Flink支持多种窗口模型比如滚动窗口，滑动窗口和会话窗口
    3. 支持实践实践、摄取时间和处理时间三种时间语义
  • Flink开发环境：Flink是以Java和Scala作为开发语言的开源大数据项目
